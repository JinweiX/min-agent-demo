# 最小 Agent Demo 技术方案规划

本文档从《AI 大航海：Claude Code 产品拆解分析 - 第二部分：技术原理》的第 12 节拆出，用作后续最小 Agent Demo 的独立方案。原文中的第 12 节保留不动，本文档用于继续细化 demo 的产品定位、体验范围、实现边界和开发阶段。

## 1. 项目定位

这个项目不是要做一个“迷你 Claude Code”，也不是要做一个马上可用于真实工作的 Agent 工具。

V0.1 定位为：

> 一个可观察的最小 Agent 机制演示器：用户输入一个简单任务后，可以在浏览器窗口中实时看到 Agent 如何理解任务、判断下一步、调用工具、吸收工具结果，并最终输出答案。

V0.1 的核心目标是理解 Agent 的基本闭环：

```text
目标输入 -> 判断下一步 -> 调用工具 -> 获得结果 -> 更新判断 -> 输出答案
```

这版 demo 的价值不在于“智能程度”，而在于把 Agent 的内部推进过程展示清楚。

真正需要回答的问题是：

- Agent 为什么不是直接回答，而是先调用工具？
- 工具结果回来后，Agent 如何改变下一步判断？
- 一个任务为什么会结束，而不是一直循环？
- 权限、安全、真实模型、多工具这些能力应该在什么阶段加入？

## 2. V0.1 产品目标

V0.1 优先级是理解 Agent 内部机制，因此先采用“假模型”驱动，而不是直接接真实大模型。

这里的“假模型”不是为了假装智能，而是为了稳定复现 Agent 的决策过程。它会按照明确规则做出下一步判断，让每一步都能被解释和验证。

V0.1 目标：

> 用户通过终端输入一个固定示例任务，系统启动一个本地浏览器观察窗口，实时展示 Agent 从收到任务、决定读取文件、获得文件内容、基于内容总结到最终结束的全过程。

标准示例任务：

```text
请读取示例工作区里的 notes.md，并总结内容
```

这个场景足够覆盖最小 Agent 闭环：

```text
收到任务
-> 发现需要文件内容
-> 调用读取文件工具
-> 获得文件内容
-> 基于文件内容生成总结
-> 输出最终结果
```

## 3. V0.1 包含什么

| 能力 | 产品意义 |
|---|---|
| 终端输入任务 | 保持最小启动入口，不先做复杂 UI |
| 示例工作区 | 提供可复现的测试文件和任务 |
| 假模型决策 | 稳定展示 Agent 为什么选择下一步 |
| Agent 执行循环 | 跑通“判断 -> 行动 -> 观察 -> 再判断”的核心机制 |
| 文件读取工具 | 用最小工具展示 Agent 如何获取外部信息 |
| 实时观察窗口 | 运行过程中看到当前执行到了哪一步 |
| 每步原因说明 | 每一步都解释“为什么这么做” |
| 完整过程回放 | 运行结束后仍可复盘全部步骤 |
| 运行记录保存 | 后续可对比真实模型版本的行为差异 |

## 4. V0.1 不包含什么

| 暂不做 | 原因 |
|---|---|
| 真实大模型 | 会引入输出不稳定，干扰理解 Agent 框架本身 |
| 多种复杂任务 | V0.1 先把一个闭环讲透 |
| 页面控制 Agent | V0.1 只观察，不做暂停、继续、打断 |
| 写文件和运行命令 | 属于危险动作，后续作为权限机制演示加入 |
| 多 Agent | 任务拆分会让 V0.1 复杂度过高 |
| 长期记忆 | 先理解单次任务闭环 |
| 插件、MCP、Hook | 属于平台化能力，不是最小闭环 |
| 复杂 UI | 页面只服务于观察执行过程，不做产品化工作台 |

## 5. V0.1 用户体验

用户体验应该是：

```text
用户在终端输入任务
↓
系统启动 Agent 运行
↓
本地浏览器打开实时观察窗口
↓
页面显示当前执行进度
↓
每一步展示“正在做什么”和“为什么这么做”
↓
文件读取结果进入上下文
↓
Agent 基于结果输出最终总结
↓
页面保留完整时间线，方便复盘
```

页面上不展示复杂技术术语，默认用产品经理能理解的表达：

| 页面展示 | 用户理解 |
|---|---|
| 当前状态 | 系统现在是在运行中、已完成还是失败 |
| 当前步骤 | Agent 现在走到哪一步了 |
| 本步动作 | Agent 准备做什么或正在做什么 |
| 原因说明 | Agent 为什么要做这一步 |
| 工具结果 | Agent 从外部拿到了什么信息 |
| 下一步变化 | 这个结果如何影响后续判断 |
| 最终回答 | Agent 最后给用户的答案 |

## 6. 实时观察窗口设计

V0.1 页面是一个“执行过程观察窗”，不是控制台。

建议页面分成 4 块：

```text
顶部：当前状态
中间：执行时间线
侧边或下方：当前步骤详情
底部：最终结果
```

### 6.1 当前状态

让用户一眼知道系统是否还在运行。

运行中示例：

```text
状态：运行中
当前步骤：正在读取文件
已完成关键节点：4 / 预期 7 个
```

完成后示例：

```text
状态：已完成
最终结果：已生成总结
```

### 6.2 执行时间线

时间线是页面核心，但它不应该是预先写死的流程动画。页面展示的每一步，都应该来自 Agent 运行中真实产生的事件。

标准示例任务预期会出现下面这些关键节点。它们用于验收和产品理解，不代表代码里要按顺序硬编码 7 步：

```text
1. 收到任务
   用户希望我读取 notes.md 并总结内容

2. 整理上下文
   我需要知道当前任务、已有信息和可用工具

3. 决定下一步
   我还不知道 notes.md 的内容，所以需要先读取文件

4. 读取文件
   正在调用文件读取工具

5. 获得结果
   已读取到 notes.md 的内容

6. 生成总结
   现在已有文件内容，可以基于内容回答用户

7. 完成任务
   输出最终总结
```

运行时当前步骤高亮，已完成步骤显示完成状态，未开始步骤保持弱化。

这里要区分两件事：

| 概念 | 含义 |
|---|---|
| 预期关键节点 | 标准示例任务正常运行时应该能看到的阶段 |
| 实际运行事件 | Agent Loop 每一轮根据上下文、决策和工具结果真实产生的记录 |

V0.1 验收看的是：实际运行事件能覆盖这些关键节点，而不是代码把这些节点按顺序播放一遍。

### 6.3 当前步骤详情

当前步骤详情回答两个问题：

- 现在在做什么？
- 为什么要做？

例如当前在“读取文件”：

```text
本步动作：
读取 examples/workspace/notes.md

为什么做这一步：
因为用户要求总结文件内容，但当前还没有文件内容。

输入：
文件路径：examples/workspace/notes.md

输出：
读取成功，获得 325 字内容
```

技术原始数据可以后续做成折叠区，不作为 V0.1 默认展示重点。

### 6.4 最终结果

任务结束后展示最终总结，同时保留完整执行时间线。

示例：

```text
最终总结：
notes.md 主要说明了……
```

## 7. Agent 最小机制

V0.1 要展示的不是复杂智能，而是 Agent 的推进方式。

可以把它理解成 6 个动作：

| 动作 | 含义 |
|---|---|
| 接收目标 | 用户告诉 Agent 要完成什么 |
| 整理已知信息 | Agent 看看现在知道什么、不知道什么 |
| 判断下一步 | Agent 判断要直接回答还是先调用工具 |
| 调用工具 | Agent 通过工具获取外部信息 |
| 吸收结果 | 工具结果回到 Agent 的上下文里 |
| 再次判断 | Agent 基于新信息继续推进或结束 |

示例任务中，关键变化是：

```text
读取文件前：
Agent 不知道 notes.md 的内容，所以不能总结。

读取文件后：
Agent 已经拿到 notes.md 的内容，所以可以总结。
```

这个变化必须在观察窗口中展示出来。否则用户只能看到结果，看不到 Agent 机制。

## 8. 假模型决策规则

假模型是 V0.1 demo 的核心。它不能只是“按顺序返回 7 个步骤”的脚本，而应该根据当前上下文做出下一步判断。

V0.1 假模型只需要支持标准示例任务，但必须体现“上下文变化会改变决策”。

建议规则：

| 当前情况 | 决策 | 原因说明 |
|---|---|---|
| 收到用户任务，但还没有整理上下文 | 先整理上下文 | 需要知道用户目标、可用工具和已有信息 |
| 用户要求总结文件，但上下文里没有目标文件内容 | 调用读取文件工具 | 没有文件内容就不能总结 |
| 读取文件成功，观察结果里已有文件内容 | 生成总结 | 已经获得回答所需信息 |
| 读取文件失败，例如文件不存在 | 输出失败说明或请求用户检查路径 | 工具没有拿到可用于总结的内容 |
| 已经生成最终回答 | 标记任务完成 | 任务闭环已经结束 |
| 连续多轮没有产生新信息 | 停止并说明无法继续 | 避免无限循环 |

这个规则表的价值在于：让 V0.1 清楚展示 Agent Loop 的本质。

```text
不是：第 1 步播放 A，第 2 步播放 B，第 3 步播放 C
而是：当前上下文缺什么 -> 判断下一步 -> 工具返回新信息 -> 再次判断
```

后续接真实模型时，真实模型只是替换这个“决策模拟器”。Agent Loop、工具调用、观察记录和实时展示机制都应该保持不变。

## 9. 系统组成

V0.1 可以按产品职责理解为 5 层：

| 层级 | 作用 |
|---|---|
| 启动入口 | 接收用户任务，启动运行流程，打开观察窗口 |
| 执行核心 | 控制 Agent 一步步推进，决定什么时候继续、什么时候结束 |
| 决策模拟器 | 用稳定规则模拟“模型决定下一步”的过程 |
| 工具区 | 提供读取文件等外部能力 |
| 观察系统 | 记录每一步，并实时推送到浏览器页面 |

技术实现上可以对应为：

| 产品职责 | 技术模块建议 |
|---|---|
| 启动入口 | CLI |
| 执行核心 | Agent Loop |
| 上下文整理 | Context Builder |
| 决策模拟器 | FakeLLM Client |
| 工具区 | Tool Registry + Workspace Tools |
| 观察记录 | Trace Recorder |
| 实时推送 | Trace Server |
| 浏览器页面 | Trace Viewer |

这里要保持一个关键边界：

> 浏览器页面只负责展示，不负责控制 Agent。

V0.1 不做暂停、继续、打断，也不在页面里做人工批准。这样能把复杂度控制在“理解 Agent 机制”这个目标内。

## 10. 实时展示的实现边界

V0.1 需要实时看到当前进度，但不需要双向控制。

因此推荐采用“单向实时推送”：

```text
Agent 运行中产生事件
-> 本地服务推送事件
-> 浏览器页面实时更新
```

技术上建议使用 SSE，也就是浏览器原生支持的单向事件流。它比 WebSocket 更适合 V0.1：

- 当前只需要 Agent 推送给页面，不需要页面反向控制 Agent；
- 实现更简单；
- 浏览器原生支持；
- 后续如果要做暂停、继续、人工批准，再升级为 WebSocket。

V0.1 实时展示只需要做到：

- 当前执行到哪一步；
- 本步为什么发生；
- 工具输入是什么；
- 工具结果是什么；
- 是否已经完成；
- 如果失败，失败在哪一步。

### 10.1 Trace Server 生命周期

实时观察窗口要明确生命周期，否则实现时容易出现“页面打开了但没数据”或“Agent 结束后页面断开”的问题。

建议规则：

| 时机 | 行为 |
|---|---|
| Agent Loop 启动前 | 先启动本地观察服务 |
| 服务启动完成 | 页面显示“等待任务开始” |
| Agent 开始运行 | 推送“收到任务”和后续运行事件 |
| 浏览器连接较晚 | 回放本次运行已经产生的完整事件 |
| 浏览器中途断开并重连 | 重新发送历史事件，再继续接收新事件 |
| Agent 正常完成 | 推送“任务完成”，保留页面可见状态 |
| 用户 `Ctrl+C` 中断 | 推送“已中断”，保存已产生的运行记录 |
| 浏览器无法自动打开 | 终端打印本地 URL，用户可手动打开 |

V0.1 不需要让观察窗口永久运行。可以采用简单策略：

- 任务运行时保持服务；
- 任务结束后保留服务一段时间，或等待用户在终端结束；
- 无论服务何时关闭，都必须先保存本次运行记录。

## 11. 运行记录格式

运行记录要和实时展示使用同一套事件结构。这样页面实时展示、运行后回放、后续对比真实模型版本，都可以复用同一份数据。

建议每次运行保存一个 JSON 文件：

```text
runs/YYYYMMDD-HHMMSS-<run-id>.json
```

建议结构：

```json
{
  "run_id": "20260617-160000-demo",
  "status": "completed",
  "user_goal": "请读取示例工作区里的 notes.md，并总结内容",
  "workspace": "examples/workspace",
  "started_at": "2026-06-17T16:00:00+08:00",
  "ended_at": "2026-06-17T16:00:05+08:00",
  "events": [
    {
      "step": 1,
      "timestamp": "2026-06-17T16:00:01+08:00",
      "phase": "decision",
      "status": "completed",
      "title": "决定下一步",
      "reason": "还没有 notes.md 的内容，需要先读取文件",
      "input": {
        "known_observations": []
      },
      "output": {
        "action": "read_file",
        "path": "notes.md"
      }
    }
  ]
}
```

字段先保持克制，V0.1 重点记录：

- 当前是第几步；
- 发生在什么时候；
- 当前阶段是什么；
- 本步是否成功；
- 本步做了什么；
- 为什么这么做；
- 输入是什么；
- 输出是什么。

## 12. Workspace 边界

标准示例可以默认使用：

```text
examples/workspace/
```

但实现时不要把路径写死。V0.1 应该支持：

```text
--workspace examples/workspace
```

默认值可以指向 `examples/workspace/`，但用户后续可以指定其他工作区。

文件读取工具必须遵守安全边界：

| 规则 | 说明 |
|---|---|
| 只能读取 workspace 内文件 | 防止 demo 意外读取项目外内容 |
| 用户传入相对路径 | 相对 workspace 解析 |
| 拒绝 `..` 逃逸路径 | 不能通过 `../` 访问 workspace 外文件 |
| 拒绝 workspace 外绝对路径 | 即使传入绝对路径，也要校验是否在 workspace 内 |
| 文件不存在时返回工具错误 | 不让程序直接崩溃 |

这条边界很重要。即使 V0.1 只是读文件，也应该让用户理解：Agent 的工具能力必须被限制在明确范围内。

## 13. 错误处理边界

V0.1 不追求复杂恢复，但必须把常见错误展示清楚。否则 demo 一旦跑不通，用户不知道问题在 Agent、工具、页面还是环境。

必须覆盖：

| 场景 | 处理方式 | 页面展示 |
|---|---|---|
| 目标文件不存在 | 工具返回失败结果，不让程序崩溃 | 显示“读取文件失败：文件不存在” |
| workspace 路径不存在 | 启动前提示错误 | 显示或输出“工作区不存在” |
| 端口被占用 | 自动尝试下一个端口，或允许通过参数指定端口 | 显示实际访问 URL |
| 浏览器无法自动打开 | 终端打印 URL | 页面可手动打开 |
| SSE 连接断开 | 浏览器自动重连并回放历史事件 | 时间线不丢失 |
| 用户 `Ctrl+C` | 优雅退出并保存已有记录 | 记录状态为 interrupted |
| 假模型无法判断下一步 | 停止并说明原因 | 显示“无法继续决策” |

这些错误处理不改变 V0.1 范围，只是让 demo 更可用、更容易排查。

## 14. 技术栈建议

V0.1 建议用 Python 实现，最低版本建议为 Python 3.10+。原因是代码量少，文件读写、命令行、JSON、简单本地服务都容易处理。

V0.1 尽量少引入依赖，优先用标准库跑通机制：

```text
argparse：命令行入口
pathlib：文件路径处理
json：结构化记录
dataclasses：定义核心数据
http.server：本地观察窗口服务
threading / queue：运行过程事件推送
unittest：最小测试
```

标准库实现 SSE 时要注意线程和服务关闭问题。V0.1 可以先接受这个复杂度，但需要在代码里明确服务关闭逻辑，避免 Agent 结束后后台线程残留。

后续再考虑引入：

```text
rich：增强终端展示
pydantic：加强结构校验
typer：优化 CLI 体验
python-dotenv：读取真实模型 API key
```

V0.1 不建议一开始就接真实模型 SDK。先证明 Agent Loop、工具调用和实时观察机制是稳定的，再替换真实模型。

阶段 1 就应该定义验证命令，而不是等到最后再补。建议先定：

```text
运行命令：python -m min_agent.cli "请读取 notes.md 并总结" --workspace examples/workspace
测试命令：python -m unittest discover -s tests
```

如果后续改用 `pytest`，需要同步更新 `AGENTS.md` 和 README。

| 目录 / 文件 | 作用 |
|---|---|
| `AGENTS.md` | 项目工作规则和验证约定 |
| `README.md` | 项目定位、运行方式和示例任务 |
| `src/min_agent/` | Agent 核心实现 |
| `web/` | 本地观察窗口页面 |
| `examples/workspace/` | 用于 demo 的示例工作区 |
| `runs/` | 保存每次运行的过程记录 |
| `docs/` | 技术方案、运行记录和后续复盘 |
| `tests/` | 最小测试 |

`trace_viewer.js` 即使不拆成多个文件，也要在代码结构上分清三类职责：

| 职责 | 说明 |
|---|---|
| 连接管理 | 连接 SSE、处理断开和重连 |
| 状态维护 | 保存事件列表、当前步骤、运行状态 |
| 页面渲染 | 根据状态更新时间线和详情区域 |

V0.1 不需要复杂前端框架，但不能把连接、状态和 DOM 更新完全混在一起。

V0.1 暂不需要 `.env.example`，因为不接真实模型，不需要 API key。等接入真实模型时再增加。

## 16. V0.1 运行流程

V0.1 运行流程可以这样理解。注意：这不是代码里的硬编码步骤，而是标准示例任务在正常情况下预期会产生的事件路径。

```text
1. 用户在终端输入示例任务
2. 系统启动本地观察窗口服务
3. 浏览器打开观察窗口
4. Agent 创建一次运行记录
5. 页面显示“收到任务”
6. Agent 整理当前已知信息
7. 页面显示“整理上下文”
8. 决策模拟器判断：需要读取文件
9. 页面显示“决定下一步：读取文件”
10. Agent 调用文件读取工具
11. 页面显示“正在读取文件”
12. 文件内容返回
13. 页面显示“获得结果”
14. Agent 基于文件内容生成总结
15. 页面显示“生成总结”
16. Agent 输出最终结果
17. 页面显示“任务完成”
18. 本地保存完整运行记录
```

这里最重要的是第 8 步和第 14 步：

- 第 8 步说明“为什么要调用工具”；
- 第 14 步说明“工具结果如何影响下一步”。

代码实现时，每一轮都应该走真实 Agent Loop：

```text
整理当前上下文
-> 根据上下文判断下一步
-> 如果需要工具，就执行工具
-> 把工具结果写回上下文
-> 再次判断
-> 直到输出最终答案或失败说明
```

## 17. V0.1 验收标准

不要用“看起来能跑”当完成标准。V0.1 至少满足：

1. 能通过终端启动标准示例任务。
2. 浏览器会打开本地观察窗口。
3. 页面能实时展示当前执行到哪一步。
4. 页面至少展示 7 个关键步骤：收到任务、整理上下文、决定下一步、读取文件、获得结果、生成总结、完成任务。
5. 每一步都有一句人能看懂的原因说明。
6. 页面能看出“读取文件前不能总结，读取文件后可以总结”的变化。
7. 最终能输出对 `notes.md` 的总结。
8. 运行结束后页面保留完整时间线。
9. 本地保存本次运行记录，方便复盘。
10. 假模型决策规则不是硬编码步骤，而是根据上下文判断下一步。
11. 文件不存在时不会崩溃，页面能展示工具错误。
12. 端口被占用时能换端口或提示可用 URL。
13. 浏览器无法自动打开时，终端能打印 URL。
14. 用户 `Ctrl+C` 中断时，已产生的运行记录能保存下来。
15. 文件读取工具只能访问 workspace 内文件。
16. 至少有 4 个测试：执行循环、上下文整理、工具注册、过程记录。

V0.1 总结质量不用追求很强，因为还没有接真实模型。重点是机制完整、过程可见、步骤可解释。

## 18. 实施阶段

建议按下面阶段推进：

| 阶段 | 目标 | 验收 |
|---|---|---|
| 1. 规则与项目骨架 | 写 `AGENTS.md`、确定目录结构、Python 版本、运行命令和测试命令 | 能运行空 CLI，测试命令可执行 |
| 2. 假模型规则 | 实现基于上下文的 FakeLLM 决策规则 | 能证明不是硬编码 7 步流程 |
| 3. 最小 Agent 闭环 | 跑通“读取文件 -> 获得结果 -> 输出总结” | 不接真实模型也能完成标准示例任务 |
| 4. 实时观察窗口 | 把运行过程实时展示到浏览器页面 | 页面能实时显示当前步骤和原因说明 |
| 5. 运行记录与生命周期 | 保存完整过程记录，处理重连、完成、中断 | 运行记录可复盘，页面不会丢事件 |
| 6. 错误处理与测试 | 覆盖文件不存在、端口占用、workspace 越界等边界 | 测试覆盖核心闭环和关键错误 |
| 7. 文档与示例完善 | 更新 README 和 runbook | 用户能按文档复现 demo |

这 7 个阶段完成后，才算 V0.1 完成。

不要把真实模型接入提前到 V0.1 中间，否则排查问题时会混在一起：

```text
到底是 Agent Loop 设计错了？
还是模型输出不稳定？
还是提示词写得不好？
还是工具参数不规范？
```

V0.1 要先把框架问题和模型问题分开。

## 一、会导致测试失败的问题

### 1. Task 10 会破坏 `test_project_structure.py` 的已有测试

[test_project_structure.py:42-46](../../tests/test_project_structure.py#L42-L46) 有一个 `test_cli_placeholder_runs_with_example_workspace` 测试：

```python
result = main(["请读取 notes.md 并总结", "--workspace", "examples/workspace"])
self.assertEqual(result, 0)
```

这个调用没有传 `--no-viewer --no-browser --step-delay 0`。Task 10 替换 CLI 后，这条测试会触发完整的 AgentLoop + TraceServer 启动 + 尝试打开浏览器。在 CI 或无显示器环境直接挂掉。而且即使有显示器，测试运行期间会弹出一个浏览器窗口。

**修法**：在 Task 10 的 Step 3 里同时修改 `test_project_structure.py`，把这条测试的调用改成：

```python
result = main([
    "请读取 notes.md 并总结",
    "--workspace", "examples/workspace",
    "--no-viewer", "--no-browser", "--step-delay", "0",
])
```

或者直接删掉它，因为 Task 10 的 `test_cli.py` 已经覆盖了更完整的 CLI 路径（包括无 viewer 模式和 workspace 校验）。

---

### 2. 最终状态判断靠中文字符串匹配，且两处实现方式不一致

`AgentLoop.run()` 里（Task 7 Step 3）：

```python
status = "failed" if message.startswith("读取文件失败") or message.startswith("无法判断") else "completed"
```

`CLI.main()` 里（Task 10 Step 3）：

```python
status = "completed" if "读取文件失败" not in answer and "无法判断" not in answer else "failed"
```

两处逻辑相同但实现方式不同（`startswith` vs `in`）。核心问题是把流程状态（成功/失败）耦合在人类可读消息上——后续换真实模型时，模型输出的中文措辞稍有变化就会导致状态判断错误。

**修法**：给 `AgentAction` 加一个 `success: bool = True` 字段（或让 `AgentLoop.run()` 返回 `tuple[str, bool]` 而不是裸字符串）。FakeLLM 在返回 `final_answer` 时设置它。CLI 直接读这个字段决定 run record 的 status，不再做字符串匹配。

具体改动涉及 3 处：

1. `types.py`：`AgentAction.final_answer()` 增加 `success` 参数
2. `agent_loop.py`：`run()` 返回 `tuple[str, bool]`
3. `cli.py`：`main()` 读 bool 决定 status

## 二、缺测试覆盖的路径

### 3. max_turns 耗尽没有测试

`AgentLoop` 的 `__init__` 接受 `max_turns` 参数并在 `for _turn in range(self.max_turns)` 之后 `return self._fail(...)`。但 Task 7 只测了成功路径和文件缺失路径，没有测"决策器永远不返回 final_answer，loop 在 max_turns 耗尽时终止"的场景。

虽然在 FakeLLM 的当前规则下这个场景不会触发（最多两轮就一定返回 final_answer），但 max_turns 是 AgentLoop 的安全机制，不应该没有测试。

**修法**：Task 7 Step 1 加一个测试：构造一个只返回 `tool_call` 的假决策器（或给 FakeLLM 一个"always_tool_call"模式），验证 loop 在达到 max_turns 后终止，event 历史里有 `run_failed`，message 包含"最大轮次"。

---

### 4. `_queues` 的线程安全靠运气

`TraceServer._broadcast()` 在主线程迭代 `list(self._queues)`（拷贝一份，安全），但 `_handle_events()` 在 daemon 线程里做 `outer._queues.append()` 和 `outer._queues.remove()`。虽然 GIL 保护了 `list.remove` 的原子性，但如果 `_broadcast` 的拷贝和 `_handle_events` 的 `remove` 恰好交错，`list(self._queues)` 的拷贝里可能包含一个马上被 remove 的 queue——这不会崩溃但会导致一个 event 被推入即将废弃的 queue（没人读了，event 丢失）。

实际影响很小（本地 demo，丢一个事件不影响体验），但应该在代码注释里标记这个已知问题。后续如果要加可靠性，再上 `threading.Lock`。

## 三、设计细节问题

### 5. FakeLLM `_summarize` 名不副实

```python
def _summarize(self, target_path: str, content: str) -> str:
    lines = [line.strip() for line in content.splitlines() if line.strip()]
    if not lines:
        return f"{target_path} 是一个空文件，没有可总结的内容。"
    preview = "；".join(lines[:3])
    return f"{target_path} 的主要内容：{preview}"
```

这不是总结，是截取前三行用分号拼接。对于多行 Markdown 文件（比如有标题、空行、列表），输出会很难读。

建议两选一：

- **保持当前实现**（V0.1 重点不在总结质量），但方法名改为 `_preview` 或 `_excerpt`，注释标明是占位实现
- **改成简单统计**：返回 `"{target_path} 共 {line_count} 行，主要内容涉及..."` 至少不产生误导

推荐第一种——诚实命名即可。

### 6. `--port 0` 的行为没有文档化

`_make_server` 里 `preferred_port == 0` 被当成"让 OS 自动分配端口"。这是测试里用的技巧，但普通用户如果传 `--port 0` 也会触发自动分配，而方案文档和 CLI `--help` 都没说明。在 argparse 里给 `--port` 加一行 `help` 说明即可。

## 四、HTML/JS 方案和现有文件的冲突

### 7. `<h1>` 缺 `id="current-title"`，实现时容易漏

现有 [trace_viewer.html:12](../../web/trace_viewer.html#L12) 是 `<h1>等待任务开始</h1>`，没有 id。方案的 JS 用 `document.querySelector("#current-title")` 选择它。方案的 Task 9 Step 3 说了要加这个 id，但没有显式标注为"修改已有元素"（和新增 HTML 片段混在一起），实现时容易漏掉导致 JS 找不到元素而静默失败。

**修法**：Task 9 Step 3 明确写"修改 `<h1>` 为 `<h1 id="current-title">`"，和新增的 HTML 片段区分开。

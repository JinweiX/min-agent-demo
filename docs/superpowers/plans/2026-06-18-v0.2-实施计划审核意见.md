# Min Agent Demo V0.2 实施计划审核意见

> 审核对象：[2026-06-18-min-agent-demo-v0.2-deepseek-plan.md](./2026-06-18-min-agent-demo-v0.2-deepseek-plan.md)
> 审核日期：2026-06-18

## 结论：方案可以进入开发，3 个小问题建议修改

V0.2 方案的设计方向正确：DeepSeek 只做决策，工具执行、安全边界、Trace 系统全留在本地。错误处理覆盖全面，测试不依赖真实 API。

---

## 一、架构评价

### 决策与执行分离

```
DeepSeek → AgentAction JSON ──→ AgentLoop ──→ ToolRegistry ──→ read_file
              ↑                      ↑              ↑
         只输出决策              只路由动作       只执行工具
```

DeepSeek 不知道 `read_file` 的实现细节，不知道 workspace 的安全边界，不知道 Trace 系统的存在。它只输出一个结构化的 `AgentAction` JSON。这个分层延续了 V0.1 的核心设计原则。

### `DecisionModel` Protocol

方案用 `Protocol` 替代 V0.1 的 `llm: object`。`FakeLLM` 和 `DeepSeekLLM` 都通过结构性子类型满足协议，不需要显式继承。`DeepSeekLLM` 又定义了自己的 `ChatCompletionClient` Protocol，让 HTTP 调用层也可替换。两层抽象各自独立，测试时可以分别 stub。

### 错误收敛路径

方案覆盖了 6 种失败模式，全部收敛为 `AgentAction.final_answer(success=False, ...)`，没有一条路径会让 CLI 崩溃：

| 失败模式 | 触发点 | 收敛路径 |
|----------|--------|----------|
| HTTP 非 2xx | `HTTPError` | `DeepSeekClient` → `ModelClientError` → `DeepSeekLLM.decide` |
| URL 错误 | `URLError` | 同上 |
| 响应非 JSON | `json.JSONDecodeError` | `DeepSeekClient.create_chat_completion` |
| 空 content | `content.strip() == ""` | `DeepSeekClient.create_chat_completion` |
| JSON 解析失败 | `json.JSONDecodeError` | `DeepSeekLLM.decide` 外层 catch |
| action 非法 | 缺少字段 / 工具不存在 / 未知 kind | `DeepSeekLLM._action_from_data` |

### API Key 安全

- 只在 `--decision-model deepseek` 时才读 `DEEPSEEK_API_KEY`
- fake 模式不碰环境变量
- `ModelClientError` 的 `__str__` 不含 key（测试 `assertNotIn("secret-key", str(caught.exception))`）
- 不加 `.env`，不写日志，不进 TraceEvent，不进 run record

## 二、需要修改的问题

### 问题 1：`--model` 参数名建议改为 `--deepseek-model`

方案定义的参数是 `--model`，argparse 存储为 `args.model`。虽然 `argparse.Namespace` 当前没有 `model` 属性，不会冲突，但如果未来 argparse 或 IDE 工具链增加同名属性，会静默覆盖。

建议改为 `--deepseek-model`，语义更明确，也避免潜在命名冲突。

### 问题 2：`build_decision_model` 应显式处理未知 `decision_model`

当前函数用 early-return 模式：

```python
def build_decision_model(args):
    if args.decision_model == "fake":
        return FakeLLM(), None
    # 隐式走到 DeepSeek 路径
    api_key = os.environ.get("DEEPSEEK_API_KEY")
    ...
```

虽然 `argparse` 的 `choices=["fake", "deepseek"]` 保证输入合法，但如果未来有人新增选项但忘记更新此函数，会静默走到 DeepSeek 路径。建议显式写 `elif args.decision_model == "deepseek"`，并在末尾加 `else` 兜底返回错误。

### 问题 3：`test_prompt_mentions_json_and_action_schema` 的断言依赖未注释

测试断言 `self.assertIn("json", system_prompt.lower())`。这个断言的正确性依赖 DeepSeek JSON Output 官方要求（prompt 中必须包含 "json" 字样）。如果未来有人修改 system prompt 但不知道这个约束，测试会失败但不知道原因。

建议在测试上方加一行注释说明来源：

```python
# DeepSeek JSON Output 要求 prompt 包含 "json" 字样
self.assertIn("json", system_prompt.lower())
```

## 三、确认无误的设计点

- `deepseek_client.py` 不依赖 `min_agent.types`，只做 HTTP + JSON，边界清晰
- `deepseek_llm.py` 对可用工具的检查是防御性的——即使漏过，`ToolRegistry.call()` 也会返回失败，双层保护
- 路径穿越（`../`、绝对路径、symlink）仍由 `resolve_inside_workspace` 拦截，DeepSeek 无论返回什么 path 都不会越界
- 所有测试用 stub/fake pattern，不需要真实 API key
- Task 执行顺序合理：Protocol → Client → LLM → CLI 集成 → 回归测试 → 文档
- V0.1 FakeLLM 保持默认，向后兼容
- 不引入 `.env`，不引入非标准库依赖

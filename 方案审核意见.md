# 方案审核意见

审核日期：2026-06-17

整体方案方向清晰，把真实模型和危险工具都推迟到后续版本、 V0.1 聚焦"可观察的机制演示"这个决策是对的。以下是具体问题和建议。

## 1. FakeLLM 决策规则完全没写——这是整个 Demo 的核心

方案里 FakeLLM 只有名字，没写它用什么规则判断"该调用工具还是直接回答"。一个只有 7 步的硬编码脚本和一组合理的决策规则，是完全不同的东西。

**建议**：在进入阶段 2 之前，先定下 FakeLLM 至少包含的规则表。比如：

| 条件 | 决策 |
|---|---|
| 上下文中没有目标文件的内容 | 调用 read_file |
| 上下文中已有目标文件的内容 | 生成总结 |
| 工具调用结果已返回 | 吸收结果，重新判断 |
| 已有最终答案 | 标记完成 |

否则写代码时大概率会变成直接 hardcode 7 个步骤的顺序执行，完全体现不出"判断-行动-观察-再判断"的闭环。

## 2. 执行时间线 7 步是预设的，不是涌现的——文档表述需要诚实

方案中 6.2 节列出了精确的 7 步时间线。但假模型如果真的"判断下一步"，步骤数量和内容取决于规则+上下文，不应该预先固定。当前写法容易让实现者直接把 7 步写成顺序脚本，跳过 Agent Loop 的核心机制。

**建议**：区分两件事——"示例任务预期会经过的关键节点"和"Agent Loop 每一步的实际输出"。7 步列表可以作为验收时的预期参考，但代码里 Agent Loop 每一步都应该真正走"上下文中有什么→判断下一步→执行"的流程。

## 3. 只覆盖了 happy path，缺错误处理设计

方案全程假设文件存在、端口可用、浏览器能打开。Demo 虽然简单，但以下场景应该处理，否则跑不起来的时候排查很痛苦：

- 文件不存在 → Agent 应该收到工具错误，并在观察窗口展示出来
- 端口被占用 → 自动换端口或用参数指定
- 浏览器无法自动打开 → 终端打印 URL，让用户手动打开
- 用户 Ctrl+C → 优雅退出，保存已产生的运行记录

## 4. Trace Server 生命周期没说清楚

三个关键问题没定义：

- **什么时候启动？** Agent Loop 启动前？同时？如果 Server 先启动但 Agent 还没开始，浏览器看到的初始状态是什么？
- **什么时候关闭？** Agent 完成后立即关闭？那浏览器还没渲染完怎么办？还是让用户手动关闭？
- **浏览器断开了怎么办？** SSE 连接断开后重连，是看到完整时间线还是只看到断开后的事件？

**建议**：Server 在 Agent 启动前启动并发送 `waiting` 状态；Agent 完成后发送 `completed` 事件并保留连接（比如 30 秒），让浏览器有时间渲染；重连时回放完整事件历史。

## 5. 运行记录格式没定

`runs/` 目录存什么格式？JSON？JSONL？一个文件还是多个？这和 `trace_recorder.py` 的实现直接相关。

**建议**：每 run 一个 JSON 文件，结构就是事件数组，每条事件包含 `{step, timestamp, type, action, reason, input, output}`。和 SSE 推送的事件结构保持一致，保存和回放可以复用同一套数据。

## 6. workspace 路径硬编码在文档里

方案里写死了 `examples/workspace/notes.md`。实现时应该做到：

- 用户可以通过 CLI 参数指定 workspace 路径
- 默认值指向 `examples/workspace/`
- 文件读取工具只能访问 workspace 内的路径（安全边界）

## 7. 缺 Python 版本声明

`dataclasses`（3.7+）、`pathlib`（3.4+）没问题，但建议声明最低 Python 版本（推荐 3.10+）。另外需要考虑 `http.server` + `threading` + SSE 长连接组合在线程清理上的痛点——Python 的 `http.server` 关闭时子线程可能残留。如果用 `asyncio` + `aiohttp` 替代，复杂度会降不少，但违背了"尽量用标准库"的原则。至少要知道这是个潜在风险。

## 8. 少了验证命令的定义

方案说"改完主动跑验证"，但 `AGENTS.md` 还没写，验证命令也没定义。阶段 4 才补测试太晚了。

**建议**：阶段 1 就应该定义测试命令（如 `python -m pytest tests/`）和运行命令（如 `python -m min_agent run "读取 notes.md 并总结"`），骨架搭好就要能跑，哪怕一开始测试是空的。

## 9. 前端文件拆了 3 个但没说明关系

`trace_viewer.html`、`.js`、`.css` 拆好了，但没说 JS 怎么管理 SSE 连接、怎么更新 DOM、需不需要状态管理。V0.1 简单，但如果 JS 里把 SSE 处理、DOM 更新、状态维护全混在一起，后面接双向控制时会很痛苦。

**建议**：V0.1 JS 至少分三层：`sse-client`（管理连接和重连）、`state`（维护步骤列表和当前状态）、`render`（根据状态更新 DOM）。

## 总结

方案的核心判断（假模型、单向推送、先不接真实 LLM）都没问题。主要风险集中在：

1. **FakeLLM 规则没写**——这是 Demo 唯一要证明的东西，不能跳过直接写死步骤
2. **错误处理和生命周期没想全**——Demo 虽然简单，但这些边界情况不处理会严重影响第一印象
3. **部分设计细节（记录格式、路径、端口）没定**——进入阶段 1 前应该补齐

建议在阶段 1（写 AGENTS.md）之前，先把上述三点定下来，再动代码。
